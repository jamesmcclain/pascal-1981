# Remediation Plan: Checklist item 8.2 (`RESET` / `REWRITE` / `GET` / `PUT`)

Status: PLANNED — not yet executed.
Audience: any agent/engineer; this document is self-contained.

## 1. Why remediation is needed

Commit `2a773cd` ("Implement core file I/O ops") marked checklist item 8.2 as
Done, but the implementation is a façade:

1. `runtime/fileops.c`: `pas_file_get` and `pas_file_put` are literal no-ops
   (`(void)f;`). `pas_file_reset` / `pas_file_rewrite` only set a mode flag.
   The `FILE *handle` field, `ensure_handle()`, and `tmpfile()` are dead code —
   nothing is ever read from or written to any handle.
2. The new test `tests/test_runtime_fixes.py::TestFileBufferModel::
   test_reset_rewrite_get_put_roundtrip` passes trivially: the value `'Z'`
   never leaves the in-memory `f^` buffer, which nothing clears. The test
   would pass with all four procedures empty.
3. Standard Pascal semantics are absent: `RESET(f)` must perform an implicit
   `GET` (load the first component into `f^`); `GET`/`PUT` must advance the
   file position; `REWRITE` must truncate to empty.
4. The checklist entry claims data "moves through the runtime file handle",
   which is false.

What *is* sound in that commit and should be kept:
- The widened FCB layout (mode word + handle slot) in `codegen/base.py` and
  `codegen/files.py`.
- The builtins-registry entries, `codegen/stmts.py` dispatch, and the
  `symbol.llvm_value is None` placeholder fix.
- Lowering the four procedures to extern runtime helper calls.

## 2. Scope and non-goals

In scope (8.2 proper):
- Real `RESET` / `REWRITE` / `GET` / `PUT` semantics over an OS-level stream,
  for both `TEXT`/ASCII files and binary `FILE OF T`.
- Lazy fill/flush of the `f^` buffer using the existing `touched` flag as the
  seam (this is exactly what 8.1's notes reserved the hook for).

Out of scope (explicitly deferred by the checklist; do NOT implement here):
- `EOF` / `EOLN` stream predicates (item 8.4). However, the FCB *should*
  record an eof bit so 8.4 has something to read — see §4.
- Filename binding `ASSIGN` / `READFN` / `CLOSE` (item 8.5). Until 8.5,
  unbound files get an anonymous `tmpfile()` backing store. This makes the
  REWRITE→PUT→RESET→GET round-trip genuinely exercisable without names.
- Attaching `INPUT`/`OUTPUT` to stdin/stdout (also 8.4/8.5 territory).

## 4. Target semantics (1981-Pascal / ISO-style core)

Define mode values: `0 = closed/unopened`, `1 = inspection (read)`,
`2 = generation (write)`. Add an eof flag (either a bit in the mode word or
reuse spare bits — recommended: keep `mode` as low 2 bits, eof as bit 2;
document the encoding in both `fileops.c` and the codegen comment).

- `REWRITE(f)`: (re)open the backing stream truncated to empty; mode :=
  generation; eof := true; buffer undefined (clear `touched`).
- `RESET(f)`: flush if previously in generation mode; seek to start; mode :=
  inspection; then perform an implicit `GET` — i.e. read the first component
  into the FCB buffer and set eof accordingly (eof := true and buffer
  undefined if the file is empty).
- `GET(f)`: requires inspection mode and not eof (on violation: print a
  diagnostic to stderr and `abort()`, consistent with `die()` style already in
  `fileops.c`). Read the next component into the buffer:
  - binary (`structure == 0`): `fread(buffer, elem_size, 1, handle)`;
    short read ⇒ eof := true.
  - ASCII/TEXT (`structure == 1`): read one `char` into the buffer. (Line
    structure / EOLN handling is 8.4; plain byte-stream chars are fine here.)
- `PUT(f)`: requires generation mode. Write the buffer:
  - binary: `fwrite(buffer, elem_size, 1, handle)`.
  - TEXT: write one `char`.
  Clear `touched` after a PUT (component consumed).
- Backing store until 8.5: lazily create with `tmpfile()` on first
  `REWRITE`/`RESET`. `RESET` on a never-opened file ⇒ empty file at eof (or
  diagnostic+abort — pick one, document it; recommended: empty-at-eof, since
  pre-ASSIGN programs can't bind names yet).

Note the asymmetry to standard Pascal: strictly, `f^ := x; PUT(f)` and
`GET(f); x := f^` is the contract; the existing buffer-variable model already
supports it. The implicit GET inside RESET is the piece most often forgotten —
do not skip it.

R4. **Make the round-trip test honest.** Replace/extend
    `test_reset_rewrite_get_put_roundtrip` so it cannot pass with no-op
    helpers:

      GET doesn't read — the in-memory buffer alone cannot produce `A` then
      `B` in that order after RESET.

    - Overwrite check: REWRITE after writing must truncate (write 2, REWRITE,

      diagnostic/abort, i.e. nonzero return code).

R5. **Add a pure-C unit check (optional but cheap).** A tiny

## 7. Risks / gotchas

- llvmlite struct GEP indices are positional — if you reorder the C struct you
  must touch `codegen/base.py`, `codegen/files.py`, and `fileops.c` together.
- `INPUT`/`OUTPUT` globals exist (`@"input" = global i8* null`). Don't make
  RESET/REWRITE on them crash differently than before; they're out of scope.
- The error path uses `abort()`; tests asserting on it should check
  `rc != 0`, not a specific signal/exit code (platform-dependent).
- Don't "improve" TEXT files with line buffering / EOLN translation here —
  that drags in 8.4 scope.
